## Problem Statement

零售投资者（包括用户自己 + 用户身边 5 个具体朋友/同事）在市场上看到价格异动或重大新闻时，没有一个简单的工具把"这件事是什么 + 价格怎么反应了 + 为什么发生"这条因果链可视化呈现出来。现有产品都让用户自己拼接：tweet 在 X，价格在 broker app，新闻在彭博，分析在 Reddit。

用户原始描述："雪球、同花顺上有新闻、有技术指标的数字，但没有关于新闻对于行情的分析，也没有 K 线/技术指标的解读。希望有一个让小白看得懂、更容易上手的平台。"

## What Makes This Cool

**Viral by design**：每个 event replay 都是一个可分享的 permalink（带 OG image preview）。当用户的朋友看完点 share，URL 进入群聊/Twitter，本身就是 growth loop。这是 TipRanks dashboard、Bloomberg Terminal 都没有的——它们是 destination，不是 artifact 生成器。

**反直觉的"哇点"**：不是给单一答案，而是展示**多空双方对同一事件的最强论点**。用户自己看到聪明人之间的分歧——这本身就是市场教育。多数 AI for finance 产品默认走"总结/答案"路线，本产品走"展示分歧"路线。

## Constraints

- 周末 ship（约 18-22 小时实际工作时间，含 demo curation 2h）
- 单人开发
- **预算上限：第一个月 ≤ $50**。Feasibility check (2026-04-29) 后实际预算 = ~$20-30（数据源 $0 via yahoo-finance2，Anthropic ≤ $20，域名 ~$10/yr）。Polygon Starter ($30/mo) 退为 Plan B。
- 必须能给 5 个具体朋友看 + 朋友能转发到群里
- 不能要求用户注册账号、不能 sub-second 响应、不能拉历史回测
- **新增（feasibility 后锁定）**：用户输入的 datetime 必须在 last 30 days 之内（yahoo-finance2 1-min 数据的硬上限）。前端表单 reject 更早的输入。

## Premises

1. 真正的 wedge 是"市场事件 → 可视化价格反应 + 多视角解读"，**Trump 推文只是 launch 时最 viral 的一个 demo 素材**。TipRanks 已经做 Trump-specific dashboard 且免费——硬碰是输。
2. 目标用户是"对市场有兴趣但工具不够用"的零售投资者——**不是纯小白**（他们看不懂时间轴），**也不是 day trader**（要 sub-second 响应，做不到）。是用户自己 + 用户身边 5 个朋友这一层。
3. MVP 周末 ship。**不训模型、不要用户账号、不做财报解析、不做归因 AI**。每多一档复杂度，ship 概率减半。
4. **差异化是"可分享的 artifact 作为 growth loop + 编辑判断哪些事件重要"**。Visualization 本身是 table stakes（TradingView/TipRanks/Bloomberg 已经在做），不是 moat。这条 premise 在 cold read 后修正。
5. "多空辩论"在 V1 就上——cold read 指出这个差异化点 ~50 行 prompt 就能实现，不应该推到 V2。

## Cross-Model Perspective

独立 Claude subagent 冷读（仅看 premises + 背景，未看对话）：

**Steelman**：A weekend-shippable, browser-only "event replay" tool. Paste any timestamp + ticker (or tweet URL, FOMC release, earnings PR, news headline) → render a clean intraday price overlay + plain-English "what likely moved and why" card. **Not a tracker, not a feed — a shareable permalink generator**. The share artifact IS the growth loop.

**Tell quote**："Builds for self + 5 specific friends/coworkers in their orbit." → 这是个 tool，不是 startup。诚实路径是建一个那 6 个人每周打开发到群里的东西，让 startup 问题用 usage 数据回答，而不是用 competitive analysis 回答。

**最大反对**：Premise 4 原版（visualization 是 moat）错了。真正的 moat 是 share artifact 作为 growth loop + 编辑判断哪些事件值得收录。证据测试：ship V1，30 天内看 5 个朋友是否外发分享了 permalink；如果零外发，可视化-as-moat 被证伪。**用户接受了这个修正**。

**未说的风险**：整个 session 优化的是"competitive landscape + 产品框架"，不是"5 个真实用户在 9:31am 想看明白市场异动时的实际行为"。pivot 链（copilot → tracker → 通用时间轴）是经典 scope drift。**正确的下一步不是写代码，而是花 30 分钟跟每个朋友聊"上次想看明白市场动作是什么时候、你实际做了什么"**。如果答案是"Google 一下就走了"，那任何可视化工具都救不了。

**用户对此的回应**：并行——这周同时 ship MVP + 跟 5 个朋友聊。

## Approaches Considered

### Approach A: Event Replay Permalink（推荐 / 选定）

单页应用。**输入 ticker + datetime → 服务端拉 ±2 小时 1-min K 线 + LLM 生成"事件是什么 + 多空两个解读" → 渲染带事件标记的 chart → 生成永久链接 `app.com/replay/abc123`，带 OG image preview**。

V1 不接受 URL 输入、不支持 A 股 / 港股 / 加密货币、不支持盘前盘后时间、不支持停牌票。这些是 V2。

- **Effort:** S（周末 ship，~18-22 小时——含 demo curation 2h）
- **Risk:** Low（除了一个数据源风险——见 Feasibility check 段）
- **Reuses:** Vercel 全套、Lightweight Charts (TradingView OSS)、shadcn/ui、Anthropic SDK

### Approach B: Curated Event Feed

A + 一个由用户编辑的"本周事件"流。Cron job + RSS/news API + 手动 curate → SEO 落地页 `app.com/this-week`。

- **Effort:** M（1-2 周）
- **Risk:** Medium（curation 是持续劳动）
- **Reuses:** A 的全套 + Supabase

### Approach C: Twitter Reply Bot

不建 destination，做 X bot。@bot 在任何 finance 推文下面 → bot 自动回复带 chart + bull/bear + permalink 的 replay。寄生在 finance Twitter 流量。

- **Effort:** M（4-5 天，X API 是大坑）
- **Risk:** High（X API 贵 + Musk 政策风险）
- **Reuses:** A 的核心引擎

## Recommended Approach: A

理由：
1. A 是 B、C 的基础——A 不行，后两者都跑不起来
2. A 验证最关键假设最便宜：30 天内 5 个朋友里有没有人外发分享 permalink？
3. B 是 A 的自然下一步：这周 ship A + 聊朋友 → 下周根据他们说哪些事件让他们困惑 → 给 A 加 curated feed 变成 B
4. C 是 Plan B：如果 A 上线 30 天没人分享 destination 链接，把同样引擎做成 reply bot 试 distribution

### Approach A 技术细节

**Stack（feasibility 后锁定）:**
- Next.js 15 (App Router) + Vercel
- Tailwind v4 + shadcn/ui
- [Lightweight Charts](https://www.tradingview.com/lightweight-charts/) 渲染 K 线
- **数据源 + 新闻：`yahoo-finance2` (Node)** —— 一个承重依赖同时拿 1-min OHLC + 头条。免费、无需 API key、纯 Node 兼容 Vercel。完整 feasibility 数据见 `feasibility/VERDICT.md`。
- Anthropic SDK，**Claude Sonnet 4.7**（最新版，cost/quality 平衡）
- Vercel KV：仅缓存 permalink → result。**不缓存价格数据**（V1 流量小，每次现拉）
- `@vercel/og` 生成 OG image：**V1 静态布局**（ticker + 大字 % 涨跌 + headline 文字），**不渲染 chart**（chart-in-OG 是 V2，Lightweight Charts 是运行时 canvas，SSR 复杂）

**Plan B**：如果 yahoo-finance2 在周末因 Yahoo 改 endpoint / 反爬而 break，迁移到 Polygon Starter ($30/月) 是 ~2 小时的工作（schema 接近，重点改 fetch 函数 + news filter）。Plan B 触发即认可 $30/mo 持续支出。

**新闻 fetch 细节**：`yahooFinance.search(ticker, { newsCount: 10, quotesCount: 0 })` 拿全局最新头条，应用层按 `providerPublishTime` 过滤到事件 ±N 小时（默认 N=4）。覆盖不到时走 step 5 的 "no headlines found" 状态，不阻塞。

**预算上限**：Anthropic ($20/mo cap) + 域名 (~$10/yr) **≤ $30 第一个月**。第二个月看用量决定是否续。

**核心数据流:**
1. 用户输入 ticker + datetime（输入框 + datetime picker；非 US 票、盘前盘后、**或超过 last 30 days** 都前端 reject）
2. 后端规范化：ticker uppercase，时间 round 到 minute，存为 ET ISO
3. 计算 hash：`sha1(ticker + rounded_minute_ET)`。先查 KV，命中直接返回（first-write-wins）
4. 未命中则 `yahooFinance.chart(ticker, { period1, period2, interval: "1m", includePrePost: false })`，**取回后客户端 filter 到 ±2h**（API 不严格 honor period2）
5. `yahooFinance.search(ticker, { newsCount: 10, quotesCount: 0 })` 拿头条，按 `providerPublishTime` filter 到事件 ±4h；空结果 → 走 "no headlines found" 状态不阻塞
6. Claude API 一次调用：返回 `{event_summary: string, bull_take: {claim: string, evidence: [{quote: string, source_url: string}]}, bear_take: same shape}`。Prompt 强制 LLM **只能引用真实头条**——没头条时返回 "insufficient news context, showing chart only"
7. 渲染：chart（事件 vertical line = 用户输入的 timestamp，不让 LLM 改）+ 三段卡片 + **每页底部固定 disclaimer**："This is not financial advice. AI-generated interpretation may be incorrect. Verify before trading."
8. 写 KV → 返回 permalink

**速率限制 / 防滥用：**
- 按 IP 限流：每 IP 每小时 ≤ 10 个新 replay
- **Anthropic 月度花费硬上限 $20**——超了直接停服（用 Anthropic 的 spend limit 或 Vercel cron 轮询 usage）
- 命中已有 permalink 不计费（KV cache hit）

**关键 UI 决策:**
- 单一输入框（ticker text input + datetime picker）
- 移动优先（朋友看到 permalink 多数在手机上点）
- 每个 replay 页面顶部明显的"复制链接"按钮
- 底部"再看一个 →"= **硬编码 5 个 demo permalink 数组**（V1 不做动态 listing）
- 全站 footer 强制显示免责声明

## Success Criteria

**周末 ship 后第 30 天检查**:
- ✅ Hard yes：**5 个朋友里 ≥2 个 *外发* 分享了 permalink**——"外发"严格定义为：他们（不是用户自己）把 permalink 发到了一个**用户不在的群/平台**。用户自己 launch 时发的 demo 不算。
- ⚠️ 边缘：5 个朋友都用了但没人外发 → 修正 distribution 策略，可能上 Approach C（reply bot）
- ❌ Fail：3 个以上朋友没回访 → 整个事件-replay 假设错了，回到用户访谈

**用户访谈的 success**:
- 5 个朋友里至少 3 个能讲出具体的"上次想看明白市场动作"故事
- 至少 1 个故事说出"我当时希望 X 但找不到"——X 就是产品方向
- 访谈实际可能 slip 到下周 Mon-Wed（约 30 分钟 × 5 + scheduling friction）。**不阻塞**周末 ship——只要求 V2 决策前完成 ≥3 个
